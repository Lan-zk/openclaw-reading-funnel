# Reading Funnel Core Goals

## 我们要做什么

我们要做一个本地、可扩展的信息漏斗内核。

它的职责不是直接替用户“读完互联网”，也不是先做一个投递产品，而是先把杂乱、重复、质量参差不齐的外部信息，稳定转化为后续流程可以继续处理的结构化候选对象。

这个项目的核心产物不是日报、消息推送或知识卡片，而是两层中间对象：

- `CandidateItem`：单条、规范化、可评分、可追踪的候选内容
- `ReadingCandidate`：由多条候选内容去重、聚合后形成的待后续处理对象

## 我们为什么做

信息过载的主要问题不是信息不够，而是判断成本太高。

每天进入视野的信息太多，来源太杂，质量差异大，重复内容多。人在真正开始精读之前，已经把大量注意力消耗在这些无效判断上：

- 这条内容值不值得点开
- 这是不是另一条内容的重复版本
- 这条信息是否噪音过大
- 这条内容是否值得进入后续处理流程

Reading Funnel Core 要解决的是这个上游问题：先把输入整理干净，再把值得继续处理的内容稳定留下来。

## 我们当前阶段要实现什么

Phase 1 只做漏斗上游预处理，不做后续编辑和投递。

当前阶段的目标是：

1. 从 `FreshRSS` 真实接入内容
2. 将来源条目落成本地原始记录
3. 将原始记录统一映射为 `CandidateItem`
4. 对候选内容做基础过滤、评分、去重和聚合
5. 生成 `ReadingCandidate`
6. 让每次运行都可追踪、可检查、可重放
7. 为每个对象定义稳定身份和明确写入归属

当前阶段明确不做：

- 消息投递
- Daily Review
- Lumina 沉淀
- 用户反馈回流
- 正文抓取
- 以 OpenClaw 为核心的业务编排

当前阶段的工程约束同时固定为：

- `FreshRSS` 只是第一个 `SourceAdapter`
- `RawSourceItem` 和 `CandidateItem` 采用跨 run 稳定逻辑身份
- `ReadingCandidate` 先采用 run 级聚类快照身份
- 每次 run 都保留原始输入快照，不依赖在线来源才能重放
- 每个持久化对象都必须有明确写入归属
- 每次 run 都必须记录规则版本和来源配置快照
- 低置信度阈值属于 `ruleset_version` 管辖范围

## 设计原则

### 先做本地内核，再做编排接入

`OpenClaw` 在后续阶段只负责消费本地产物并执行编排。它不是当前 Phase 1 的业务中心。

### 先保留来源特性，再做统一抽象

系统的核心模型不是 `FreshRSSItem`，而是：

`SourceAdapter -> RawSourceItem -> CandidateItem -> ReadingCandidate`

`FreshRSS` 只是第一个 `SourceAdapter` 实现，不是永久唯一入口。

### 先做可继续处理的中间层，不提前做终局产品

当前阶段的目标不是生成最终内容，而是为后续精选、人工判断、沉淀和投递准备稳定输入。

### 先保证边界清晰，再考虑多 Agent 拆分

我们会按多 Agent 友好的边界来设计模块，但当前阶段不强制一开始就做成全多 Agent 运行时。

## 当前阶段完成标志

当下面几件事都成立时，Phase 1 的目标才算达成：

- 系统能从 `FreshRSS` 拉到真实数据
- 系统能产出 `CandidateItem`
- 系统能产出 `ReadingCandidate`
- 相同来源条目在不同 run 中复用相同逻辑身份
- 聚合结果可追踪到原始来源
- 原始条目、候选条目和聚类条目都有明确写入归属
- run 失败记录和规范化失败记录都有明确契约
- 失败、过滤和低置信度情况是可见的
- 系统能从历史原始快照重放一次 run
- 下游只默认消费 `SUCCEEDED` 或 `PARTIAL_SUCCESS` 的 run
- 本地产物可以被后续 agent 或流程继续消费
